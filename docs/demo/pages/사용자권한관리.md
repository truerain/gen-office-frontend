# 사용자별 권한 관리 페이지 설계

## 1. 목적
- 운영자가 사용자-권한 매핑(`userId + roleId`)을 조회/등록/수정/비활성화하는 관리자 화면을 제공한다.
- 백엔드 API 문서 `backend/docs/api/admin/사용자권한관리.md`를 기준으로 프론트 CRUD 동작을 설계한다.

## 2. 메뉴/라우팅 구조 (DATA_DRIVEN_MENU 기준)
- 메뉴 데이터 소스: `/api/app-menus`
- 페이지 연결 우선순위:
  - 1순위: DB `execComponent`
  - 2순위: 프론트 폴백 매핑(`menuData.ts > componentNameByMenuId`)
- 페이지 컴포넌트 이름(권장): `UserRoleManagementPage`
- 동적 로딩 등록 파일: `apps/demo/src/app/config/componentRegistry.dynamic.ts`

## 3. 운영 반영 시 필수 설정
### 3.1 DB 메뉴 설정
- 테이블: `app_menus`
- 권장 값:
  - `menu_name`: User Role Management
  - `parent_menu_id`: System 메뉴 ID
  - `exec_component`: `UserRoleManagementPage`
  - `display_yn`: `Y`
  - `use_yn`: `Y`
  - `sort_order`: 운영 정책에 맞게 설정

### 3.2 프론트 폴백 매핑
- 파일: `apps/demo/src/app/menu/menuData.ts`
- 목적: DB `execComponent` 누락 시에도 진입 보장
- 권장 매핑:
  - `900700 -> UserRoleManagementPage` (신규 메뉴 ID는 운영 정책에 맞춰 조정)
  - 아이콘: `UserCog` 또는 `ShieldCheck`

## 4. 권한/접근 정책
- 관리자 권한(예: `ROLE_ADMIN`)만 접근 가능해야 한다.
- 운영 점검 항목:
  - 비관리자 접근 시 401/403 차단
  - 세션 만료 시 표준 에러 응답 노출

## 5. 화면 기능 설계
- 경로(예정): `apps/demo/src/pages/admin/user-role/UserRoleManagementPage.tsx`
- UI 구조:
  - 상단: 검색 필터 바
  - 본문: 단일 그리드(`GenGridCrud`)로 사용자-권한 매핑 목록/편집
- 검색 조건:
  - `userId` (숫자)
  - `roleId` (숫자)
  - `useYn` (`Y`/`N`)
- 주요 기능:
  - 목록 조회 (`GET /api/mis/admin/user-roles`)
  - 행 추가/수정/삭제 일괄 저장
  - 사용자/권한 표시 정보(`empNo`, `empName`, `orgName`, `roleName`) 읽기 전용 노출
  - 삭제는 물리 삭제가 아닌 비활성화(`useYn = 'N'`) 처리

## 6. API 연동 설계
- 목록: `GET /api/mis/admin/user-roles`
  - query: `userId`, `roleId`, `useYn`, `sort`
- 단건: `GET /api/mis/admin/user-roles/{userId}/{roleId}`
- 생성: `POST /api/mis/admin/user-roles`
- 수정: `PUT /api/mis/admin/user-roles/{userId}/{roleId}`
- 삭제(비활성): `DELETE /api/mis/admin/user-roles/{userId}/{roleId}`

## 7. 프론트 모델/타입 설계
- 파일: `apps/demo/src/entities/system/user-role/model/types.ts`
- 권장 타입:
  - `UserRoleKey`: `userId`, `roleId`
  - `UserRole`: `primaryYn`, `useYn`, `attribute1~attribute10`, `createdAt`, `updatedAt`, `empNo`, `empName`, `orgName`, `roleName`
  - `UserRoleListParams`: `userId?`, `roleId?`, `useYn?`, `sort?`, `page?`, `size?`
  - `UserRoleCreateRequest`
  - `UserRoleUpdateRequest`

## 8. API/Query 레이어 설계
- 파일: `apps/demo/src/entities/system/user-role/api/userRole.ts`
- 구현 항목:
  - `userRoleApi.list(params)`
  - `userRoleApi.get(userId, roleId)`
  - `userRoleApi.create(body)`
  - `userRoleApi.update(userId, roleId, body)`
  - `userRoleApi.remove(userId, roleId)`
  - `useUserRoleListQuery(params)`
- 응답 파싱:
  - 배열 응답(`UserRole[]`) 우선 대응
  - 필요 시 `{ items }` 래핑 응답도 함께 방어적으로 지원

- `apps/demo/src/pages/admin/user-role/UserRoleManagementPage.tsx`

- `apps/demo/src/pages/admin/user-role/UserRoleManagementPage.module.css`

## 10. 그리드 컬럼 설계
- 필수 노출:
  - `userId` (숫자, 생성 시 수정 가능 / 수정 시 비수정 권장)
  - `roleId` (숫자, 생성 시 수정 가능 / 수정 시 비수정 권장)
  - `primaryYn` (select: `Y`, `N`)
  - `useYn` (select: `Y`, `N`)
  - `empNo`, `empName`, `orgName`, `roleName` (read-only)
  - `attribute1~attribute10` (text)
  - `updatedAt` (read-only)
- UX 권장:
  - `userId`, `roleId` 좌측 pin
  - `useYn` 중앙 정렬
  - `Y/N` 선택 옵션 고정

## 11. 커밋 로직 설계 (`UserRoleManagementCrud.ts`)
- 변경 수집:
  - `create`, `update`, `delete`, `undelete` 분리 수집
- 처리 순서:
  1. 생성(create): `POST`
  2. 수정(update): `PUT`
  3. 삭제(delete): `DELETE` (서버에서 `useYn='N'` 처리)
- 복합키 주의사항:
  - 기존 행의 `userId/roleId` 변경은 신규 생성+기존 삭제로 취급하거나 UI에서 키 편집을 막는다.
  - 안정성을 위해 키 편집 금지를 기본 정책으로 권장한다.

## 12. 입력 검증/에러 처리
- 필수값:
  - `userId`, `roleId`, `primaryYn`, `useYn`
- 규칙:
  - `userId`, `roleId`는 0보다 큰 정수
  - `primaryYn`은 `Y` 또는 `N`
  - `useYn`은 `Y` 또는 `N`
  - 동일 사용자 기준 `primaryYn='Y'`는 1건만 유지(백엔드 정책)
- 에러 처리:
  - `409 Conflict`: "이미 존재하는 사용자-권한 매핑입니다."
  - `404 Not Found`: 사용자/권한 미존재 메시지 노출
  - 기타 오류: 공통 알림 + 서버 메시지 우선 표시

## 13. 화면 액션/동작 정책
- 액션바:
  - 기본: `add`, `delete`, `save`, `filter`
  - 커스텀: `refresh`
- 저장 전:
  - 필수값 누락 검증
  - 저장 확인 다이얼로그
- 저장 후:
  - 목록 재조회(`refetch` 또는 query invalidate)
  - 성공 알림 표시

## 14. 장애 점검 체크리스트
1. 메뉴는 보이는데 페이지가 안 열림
- `/api/app-menus`의 `execComponent` 확인
- `componentRegistry.dynamic.ts` 등록 여부 확인
- `menuData.ts` 폴백 매핑 확인

2. 페이지는 열리는데 목록이 비정상
- `/api/mis/admin/user-roles` 상태코드/응답 확인
- 검색 파라미터(`userId`, `roleId`, `useYn`) 전달 확인
- 권한 오류(401/403) 확인

3. 저장 시 실패
- 중복 키(409) 여부 확인
- 필수값/형식(`Y|N`, 정수 ID) 확인
- 서버 에러 포맷(`code`, `messageKey`, `message`) 확인

## 15. 배포 후 검증 시나리오
1. 관리자 계정으로 System > User Role Management 진입
2. `userId`, `roleId` 조건으로 조회 확인
3. 신규 매핑 1건 생성 후 재조회 확인
4. 동일 키 재생성 시 409 메시지 확인
5. `useYn`/attribute 수정 후 반영 확인
6. 삭제 액션 수행 후 해당 매핑이 비활성(`useYn='N'`) 처리되는지 확인
